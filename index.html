Lectures 1 – 9

Foundations

What software architecture is, why it matters, and the architect's role

§1–3 · What Is Software?

Software encompasses programs, source code, documentation, data, tools, procedures, and version control configurations. A software-intensive system is one where software is the major technical challenge and primary factor affecting schedule, cost, and risk.

DefinitionA system is a combination of interacting elements organized to achieve one or more stated purposes.

§4–6 · Architecture Principles & Goals

From Vitruvius: architecture must be useful (functional), stable & reliable (proven solutions), and graceful (well-structured, great UX).

Goals of software architecture:

  • Create meaningful structure — decompose via patterns and best practices
  • Specify key implementation details — help developers execute correctly
  • Define rules, principles, and guidelines — conceptual integrity
  • Document design decisions and rationale — including rejected alternatives

§7 · Complexity

DefinitionComplexity is the human effort required to understand, change, or extend a system. Once added, it is very difficult to remove and accumulates until the system becomes impossible to change.

Robert C. Martin's goal: minimize the human resources required to build and maintain the system.

§8–9 · Project vs. Architecture Goals

Project Goals (Short-Term)Architecture Goals (Long-Term)
Deliver features on timeEnsure long-term maintainability
Stay within budgetReduce complexity over time
High quality for current releaseSupport extensibility & understandability
Short-term operational focusInvest in infrastructure & technology
The ConflictShortest time and smallest budget conflicts with long-term architecture health. Both extremes are dangerous. The architect's job is to make smart trade-offs.

§10 · Defining Software Architecture

No single universal definition, but all share four elements:

Principles, Rules & GuidelinesSystem-wide design governance (often overlooked)

§11 · Components

A component is a unit of composition and encapsulation providing static structure. Can be any size — from a function to an entire container cluster.

Black BoxWhite Box
Hides internals; exposes only I/O & responsibilities. Reduces complexity.Shows internal structure. Used for internal design documentation.

§12–13 · Interfaces

An interface is a contract and shared boundary — covering syntax, functional behavior, error behavior, quality attributes, technologies/protocols, and constraints.

Provided (API)Required (SPI)Standard
Provider defines & owns. Consumer depends on provider.Consumer defines. Provider must conform. Dependency inverted.Third party defines. Both conform.

§14–18 · Stakeholders, Requirements & Quality

Stakeholders express concerns through requirements — the bridge between people and architecture.

Functional RequirementsQuality Requirements
WHAT the system does. Changes usually localized.HOW the system behaves. Missing quality reqs → major redesign.

Key Quality Attributes

  • Performance — throughput (work/time) and/or response time (latency)
  • Efficiency — work done per resource used; critical for IoT/battery devices
  • Scalability — potential to increase performance: vertical (scale up), horizontal (scale out), elastic (auto add/remove)
Key RuleA system can be scalable without being elastic. But elastic scaling requires horizontal scalability as a prerequisite.

Constraints

Externally imposed limitations (technical, time/budget, legal/regulatory) that narrow the solution space — not always bad, as they reduce decisions the architect must make.

Lectures 4 – 5

Tasks & Responsibilities

The seven core tasks of a software architect

Task 1 — Establish the Business Case

Create economically viable solutions. Understand cost, target market, time to market, product lifetime, and integration needs. The architect contributes input, not owns it.

Task 2 — Clarify & Scrutinize Requirements

Guarantee fulfillment by identifying, clarifying, refining, and scrutinizing all requirements. Key sub-tasks:

  • Feasibility — Is it technically achievable and economically reasonable?
  • Contradictions — Detect conflicts (e.g., absolute privacy vs. law-enforcement access)
  • Implicit requirements — Make hidden expectations explicit (e.g., adaptive streaming)
  • Unstable requirements — Flag frequently changing reqs; stabilize ASAP to avoid scope creep
NoteThe architect is not the requirements engineer. The architect guarantees fulfillment — not definition.

Task 3 — Design or Select the Architecture

Select existing, extend current, or design from scratch — weighing cost, integration, and alignment. Technology selection: prefer proven tech, consider developer experience, evaluate landscape. Decompose into building blocks, define interfaces, address cross-cutting concerns. Document rationale for every decision. Prototype before handoff.

Task 4 — Document & Communicate

Create visual, illustrative presentations (diagrams, views, slide decks). Always document assumptions explicitly to prevent misunderstandings.

Task 5 — Guide & Support Developers

Not an ivory-tower role. Stay involved: ensure compliance, conduct code reviews, gather feedback, support tooling, analyze issues, coordinate cross-project. Learn from skilled developers.

Task 6 — Evaluate the Architecture

Quantitative (metrics for performance, scalability, reliability, complexity) and qualitative (adherence to requirements, constraints, rules).

Task 7 — Identify & Mitigate Risks

Identify risks around quality requirements, communicate consequences, propose mitigation strategies. Example: healthcare system on public cloud — justify compliance, propose backup redundancy.

KeyThese tasks are not linear — the architect iteratively switches between them throughout the project lifecycle.
Lecture 6

Skills of a Software Architect

Technical depth, soft skills, and the growth mindset

Technical Skills

Foundation of the role. Not the fastest coder, but solid ability to evaluate solutions, build prototypes, and earn developer trust. Alone insufficient — many talented developers struggle as architects.

Soft Skills

  • Organisation — structure work, track decisions, manage documentation
  • Negotiation — navigate competing stakeholder interests
  • Communication — convey complex ideas to technical and non-technical audiences
  • Listening — understand needs, concerns, constraints
  • Guiding others — mentor developers toward architectural vision

Abstract & System-Level Thinking

Step back from code and see the big picture. Biggest hurdle for skilled coders transitioning to architect. Involves reasoning without knowing every detail, identifying patterns across codebases, and anticipating emergent behavior.

Decision-Making

Structured evaluation → accept uncertainty → document rationale → be willing to revisit. Indecisiveness is incompatible with the role.

Breadth Over Depth

Wide awareness across many domains. Know enough to ask the right questions and delegate with confidence. Avoid going so deep you lose the big picture.

MindsetNot all skills are required from day one. Self-awareness, deliberate practice, and patience — excellence is a long-term journey.

Conway's Law & Inverse Conway Maneuver

Conway's Law"Organizations which design systems are constrained to produce designs which are copies of the communication structures of those organizations."

Inverse Conway Maneuver: deliberately structure teams to match the desired target architecture. Organization → Architecture becomes Architecture → Organization.

Three Categories of Influencing Factors

Product-RelatedOrganizationalTechnological
Business goals, quality reqs, legal/regulatory compliance, industry standardsCompany structure, team skills, budget, timeline, culture, processesExisting infrastructure, integration protocols, available tools, team expertise
Sweet SpotThe best architecture is not always the most technically elegant — it's the one that best balances all influencing factors.

System Context

The part of reality relevant to the system's requirements: stakeholders, existing systems, events, processes, and documents.

Legal Factors

Privacy, healthcare data, children's online privacy, data sovereignty. The architect doesn't need to be a legal expert but must understand implications for storage, retention, and technology choices.

Lectures 9 – 10

Quality Requirements & Risk

Quality models, trade-offs, and risk management

Quality Attributes vs. Goals vs. Requirements

  • Quality Attributes — the properties themselves (performance, availability, security…)
  • Quality Goals — high-level objectives involving multiple attributes ("improve UX")
  • Quality Requirements — specific, actionable, measurable ("99.9% uptime", "page load ≤ 500ms")

Relationships Between Quality Attributes

Positively CorrelatedNegatively Correlated
Performance ↔ ScalabilityMemory usage ↔ CPU (caching)
Simplicity ↔ UnderstandabilitySecurity ↔ Usability (MFA)
Reliability ↔ AvailabilityPerformance optimization ↔ Flexibility

Risk Management

FormulaRisk Severity = Probability × Amount of Harm

Common Risk Categories

  • Schedule & Scope — tight deadlines → technical debt; missing MVP definition → scope creep
  • Requirements — ambiguity, high change rate, poor prioritization
  • Documentation — absent docs are costly and underrated
  • Infrastructure — underestimated resources fail at peak load
  • Expertise — single specialists are single points of failure
  • Technology — untested products introduce uncertainty
  • External Interfaces — late third-party changes force rework

Mitigation Tips

  • Define MVP and buffer the schedule. Compromise on scope, not quality.
  • Adding people to a late project makes it later.
  • Apply Murphy's Law — plan worst-case plus safety margin.
  • Make all implicit assumptions explicit (including what a component is not responsible for).
Lectures 11 – 18

Design Principles

From information hiding to SOLID

§66–67 · What Are Design Principles?

Distilled knowledge guiding design decisions. Two objectives: reduce complexity and achieve long-term quality. Principle #1: know when to break the rules — fulfil requirements first, principles are tools not laws.

§68 · Information Hiding

Treat every component as a black box. Hide internal details; allow access only through defined interfaces. Applies at all abstraction levels. Violations create tight coupling and fragility.

§69 · Separation of Concerns

Each component handles one distinct concern. Mixing concerns increases coupling, complicates testing, reduces independent changeability. Complementary to information hiding.

§70 · Modularity

Self-contained modules with defined interfaces. Enables independent development, testing, reuse, and composition. The structural expression of information hiding + separation of concerns.

§71–73 · Loose Coupling

Keep dependencies few and simple. Seven coupling types from tightest to loosest:

TypeTightness
InheritanceStrongest — parent changes ripple to all children
Caller–CalleeVery tight — direct method call
CreationTight — instantiates and manages lifecycle
Shared Data StructureModerate — shared map/DB/file
Time Coupling (sequential, scheduling, real-time)Varies
Execution LocationTight — depends on specific env
Message ExchangeLooser — queue/broker decouples sender from receiver

§74–75 · High Cohesion

Degree to which elements inside a component serve a unified purpose. High cohesion → low coupling. Context-dependent and hard to measure precisely.

§76 · KISS — Keep It Simple

Simple systems are easier to understand, maintain, and less error-prone. Tension: simplicity vs. future flexibility, simplicity vs. performance.

§77 · YAGNI — You Aren't Gonna Need It

Don't implement functionality before it's actually required. Unused code must still be read, understood, tested, and maintained.

§78 · DRY — Don't Repeat Yourself

Single authoritative representation. Eliminates duplicate work, bugs, and tests. Trade-off: DRY introduces coupling — in microservices, duplication may be preferable to shared coupling.

§79–80 · Expect Errors & Robustness Principle

Anticipate failure modes at design time. Contain blast radius (circuit breakers, fallbacks). Support analysis (logs, metrics, tracing).

Postel's LawBe strict in what you produce; be liberal in what you accept. Tolerate reasonable input variation; degrade gracefully.

§81–82 · Abstraction & Conceptual Integrity

Create generalisations that hide implementation detail. Depend only on abstractions, never concrete implementations. Conceptual integrity: build the whole system around a consistent set of clear concepts — the most important consideration in design.

§83–87 · SOLID Principles

Lectures 19 – 24

Architecture Patterns

Layers, Pipes & Filters, Microservices, DI, and more

§88 · What Are Patterns?

Reusable solutions to recurring problems — discovered, not invented. Libraries reuse code; frameworks reuse structure; patterns reuse thinking.

§89–92 · Layers Pattern

Vertical stack: each layer depends only downward, provides abstraction upward.

Closed (Strict)Open (Loose)
Call only next layer down. High modularity, predictable.Skip layers. Better performance, more coupling.

Layer vs. Tier: Layers = logical structure; Tiers = physical deployment. Both can coexist.

§93–95 · Pipes & Filters

Data flows through a pipeline: source → filter (transform) → pipe (transfer/buffer) → … → sink. Filters are independent and unaware of each other.

Use cases: multimedia encoding, Unix CLI pipelines, streaming VOD ingestion, ETL pipelines.

Pros: simplicity, flexibility, scalability, team scalability. Cons: error propagation harder, buffer overflow risk, poor for shared state.

§96–98 · Microservices

Small, independently deployable units each responsible for a single business capability. Key practices:

  • Single Responsibility — one business domain per service
  • Database per Service — no shared schemas; data duplication is accepted overhead
  • Inverse Conway Maneuver — small focused teams → loosely coupled architecture
ProsCons
Independent change, scale, deployNetwork latency, outages, bandwidth
Tech flexibility per serviceError tracing across services harder
Faster dev (smaller codebases)Operations complexity much higher
Runtime errors (not compile-time)

§99–101 · Dependency Injection

A DI framework creates concrete implementations at runtime and injects them into classes that declare a dependency on an abstraction.

Key terms: assembler/injector, bean/component, injection point (constructor/setter), configuration.

IoC vs. DI: DI is a special case of Inversion of Control. All DI is IoC, but not all IoC is DI (callbacks, schedulers, event loops are also IoC).

§118–121 · Reducing Coupling Tactics

Structural Dependency

  • Polymorphism — depend on interface, not concrete class
  • Facade — single entry-point shielding internal complexity
  • Decorator — layered behaviors via delegation; no coupling between decorators

Instantiation Dependency

  • Factory Pattern — centralized creation; clients depend only on interface
  • Proxy Pattern — lazy initialization; same interface wrapper
  • Dependency Injection — framework wires entire dependency graph

Call Dependency

  • Abstractions / Adapter — interface between caller and callee
  • Avoid cycles — cycles cause tight coupling and unpredictable behavior
  • Async messaging — fire-and-forget via broker; decouples from receiver's availability
  • Restrict communication — layer rules contain coupling within boundaries

§121 · Static vs. Dynamic Coupling

AspectStaticDynamic
Determined atCompile timeRuntime
PerformanceHigher (no resolution overhead)Lower
ToolingFull IDE + compiler supportLimited; runtime tools only
FlexibilityLow (rebuild to change)High (reconfigure without rebuild)
DebuggabilityEasy (errors caught early)Hard (errors surface in production)
CriticalDynamic ≠ loose. Switching to dynamic coupling does not reduce coupling. DIP must also be applied — inject abstractions, not concretes.
Lectures 25 – 28

Cross-Cutting Concerns & Interfaces

From AOP to interface best practices

Cross-Cutting Concerns

Aspects that affect multiple components: persistence, error handling, security, internationalization, communication, UI/UX, business rules. Without management: DRY violations, reduced conceptual integrity, high change cost.

Annotations & AOP

Annotations — metadata decorators evaluated at compile time (code gen, static analysis) or runtime (DI, ORM).

Aspect-Oriented Programming — separates cross-cutting concerns from business logic; weaves them in automatically. Key terms: aspect (encapsulated concern), advice (action: before/after/around), pointcut (which join points), join point (method execution).

Interface Design

Resource-Oriented (REST)Service-Oriented (RPC)
Fixed uniform operations on URI-identified resourcesArbitrary domain-specific operations
Lower complexity, multiple representationsMore expressive, higher complexity

Sync vs. Async: sync blocks until response; async continues, result delivered later.

Interface Best Practices

  • Try using the interface yourself (build test clients)
  • Clarify requirements with both consumer and provider
  • Apply information hiding — expose only what's needed
  • Meaningful, self-descriptive names; include units for quantities
  • Minimize surprises and side effects
  • Provide atomic information units (no blob parsing)
  • Write thorough documentation

Product & Sum Types

Product type: all attributes required simultaneously (AND). Sum type: exactly one variant active (OR — discriminated union). Relational DBs and JSON can't natively enforce sum types.

PrincipleDefine the data model abstractly — each storage/protocol layer encodes it independently. Enables independent optimization and migration per layer.
Lectures 29 – 31

Design Strategies

Achieving performance, flexibility, and high availability

High Performance

  • Add hardware — more servers, memory, faster CPUs
  • Reduce communication — cache frequently accessed data; batch multiple requests
  • Adjust distribution — parallelize queries but avoid excessive cross-instance coordination
  • Reduce flexibility — hard-code stable business logic to eliminate configuration hops
  • Continuous load testing — optimizations erode; test before high-traffic events
  • Compromise energy efficiency — last resort (disable throttling, add GPUs)

Adaptability & Flexibility

  • Target flexibility where needed — Strategy pattern, feature flags, facades, responsive design, containerization
  • Configuration files — externalize environment settings
  • Keep changes local — high cohesion + single responsibility
  • Polymorphism — common interface enables adding/changing implementations
  • Information hiding — details change without affecting clients
  • Reduce coupling — limits change propagation
  • Maintainable code — prerequisite for adaptability

High Availability

1. Error Prevention

Transactions (atomicity), input validation (reject at boundary), back pressure (prevent queue overflow).

2. Error Detection

Continuous monitoring (metrics dashboards), cross-validation (independent reconciliation).

3. Error Handling

Exception handling & retry, rollback (data + deployments), redundancy (load balancer + replicas), automatic replacement (health monitoring restarts instances).

Architecture Development & Views

Iterative design, arc42 views, top-down vs. bottom-up

Iterative & Incremental

Iterative: repeat design steps to refine decisions and incorporate feedback. Incremental: each iteration extends the solution by a focused set of concerns. Architecture feedback is not automatic — it must be actively sought through structured stakeholder reviews.

Views may overlap but must be consistent and never contradict each other.

Top-Down vs. Bottom-Up

DimensionTop-DownBottom-Up
Starting pointComplete system vision; decomposeIndividual components; assemble
Best forLarge-scale, clear scopeR&D-heavy, parallel dev
Integration riskHigher (assumptions may not hold)Lower (details resolved first)

Both can be combined: start bottom-up for ambiguous parts, then apply top-down for the rest.

Documentation & Evaluation

Templates, ADRs, diagrams, metrics, ATAM

Why Document?

Source code is not documentation — it lacks structural overview, decision rationale, technology context, and cross-cutting concerns. Documentation cost directly impacts maintenance, evolution, and bug-fixing speed.

PrincipleVerbal communication leads to decisions. Written communication records them.

Architecture Decision Records (ADRs)

Lightweight format: Title → Status (under review / accepted / deprecated / superseded) → Context → Decision + Rationale → Consequences (positive & negative).

Documentation Best Practices

  • Use a standardized structure or template
  • Write from the reader's perspective (active voice, simple language)
  • Avoid unnecessary repetition
  • Avoid ambiguity (formal notation, consistent names, glossary)
  • Provide rationale for decisions; record rejected alternatives
  • Declutter diagrams: 7 ± 2 elements per diagram
  • Keep documentation current (fixed sync cadence, not every minor change)
  • Communicate status (change log, version control, stability signals)
  • Review before publishing with target audience representatives

Quality Evaluation

Static AnalysisDynamic Analysis
WhenWithout executing softwareDuring execution
WhatLOC, coupling, complexity, effort to changeResponse times, resource usage, test results

Key Metrics

  • Instability: I = Ce / (Ca + Ce). I=0: stable; I=1: unstable
  • Cyclomatic complexity: number of independent execution paths
  • MTBF / MTTR: reliability indicators
Goodhart's LawWhen a measure becomes a target, it ceases to be a good measure. Use metrics as diagnostic signals, not mandatory targets.

ATAM — Architecture Trade-off Analysis Method

Leading qualitative evaluation method (Carnegie Mellon SEI). Three focuses: early risk identification, making correct trade-offs, identifying sensitivity points.

Process: business drivers → quality attributes → scenarios → utility tree (prioritized) → architect presents → trade-off analysis → sensitivity points → risk themes feed back.

Benefits: clarifies quality reqs, improves docs, documents decisions, early risk detection, better stakeholder communication.
